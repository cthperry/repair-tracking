## V161.303 補充原則
- 維修詳情頁 `總覽` 必須先呈現可直接判讀的案件總覽，不可把核心資訊拆散到左主欄與右側輔助卡再由使用者自行拼湊。
- 同一份總覽中不得出現重複資訊卡；狀態、收費、設備、客戶資料應各自落在固定語意區塊。
- 問題描述、補充說明、內部備註屬於同一閱讀脈絡，應留在主閱讀區，避免切成多個零散卡片。

## V161.302 補充原則
- 大型表單首屏必須呈現模式、關聯對象與關鍵狀態，不可讓使用者先讀完整個 modal 才理解自己在編輯什麼。
- 大型表單 modal 一律優先採 `header + scroll body + sticky footer`。
- 只讀欄位不得沿用可選欄位元件；若資料已由上層流程決定，應以鎖定/唯讀語意呈現。
- 關鍵主資料欄位（公司名稱、Email、問題摘要等）優先放在首屏主閱讀區。

# UI / UX Guideline

本文件作為 RepairTracking 後續 UI 收斂基準。

## 1. 設計原則
1. 先確保資訊可讀，再做視覺強化
2. 桌機與手機必須同時成立
3. 互動安全優先，避免誤觸、穿透點擊、遮擋主要按鈕
4. 同一類元件只能有一套主要規則

## 2. 視覺基底
依 `core/config.js` 與既有 UI 規格整理：
- 背景：淺色、低彩度
- 層次：`background → panel/surface → card`
- 主色：`primary = #2563eb`
- 成功：`#16a34a`
- 警示：`#d97706`
- 錯誤：`#dc2626`

字體建議：
- `Noto Sans TC`
- `Microsoft JhengHei`
- sans-serif fallback

## 3. 版面規則
### Desktop
- 主內容區採單一主要閱讀流
- 輔助資訊可放側欄，但不得形成過長難讀的右側資訊柱
- Dashboard / Detail / WorkLog 的 KPI、摘要、操作列需一致化

### Mobile
- 所有主要操作需落在單手可及區
- 固定底部區塊必須預留 safe-area
- 表單與彈窗不得遮住主要儲存按鈕
- 長列表點擊區需避免與滑動手勢衝突

## 4. FAB 規範
目前已落地規則：
- `settings` / `guide` / `sops` 頁隱藏
- modal 開啟時隱藏
- 接近頁尾或碰撞底部操作區時避讓或隱藏

後續固定規範：
1. FAB 不得遮擋 sticky footer
2. FAB 與浮動面板需共用同一安全邊界
3. Mobile 必須考慮瀏覽器底部工具列與 safe-area inset

## 5. Modal 規範
1. 標題、主內容、操作列三段式固定
2. 主操作按鈕放右側，取消放左側
3. 表單 modal 內容區要可滾動，但操作列固定可見
4. 不得讓 modal 內主要按鈕被 FAB、footer 或瀏覽器工具列遮住

## 6. Sticky Footer / Save Bar 規範
1. 僅在有主要提交流程時顯示
2. 不得覆蓋表單最後一個輸入欄
3. 與 FAB 之間要有明確避讓距離
4. 按鈕文案固定：`儲存` / `建立` / `取消` / `返回`

## 7. Badge / Status 規範
### Repair
- `進行中`
- `需要零件`
- `已完成`

### Quote
- `草稿`
- `已送出`
- `已核准`
- `已取消`

### Order
- `建立`
- `已下單`
- `已到貨`
- `已結案`
- `已取消`

### 顯示原則
1. 顏色與文字必須固定對應
2. 不可同義多名
3. badge 顏色只承載狀態，不承載優先級以外訊息

## 8. Detail / WorkLog 規範
### Repair Detail
- 以「主流程」為主：問題 → 處置 → 商務 → 零件 → 關聯資料
- 避免右側長柱式資訊密集堆疊
- 同一頁最多一個主要 CTA 群組

### WorkLog
- 以時間序 + 結果狀態為核心
- 每筆卡片需清楚分出：日期、工作內容、發現、零件、結果
- 手機版優先考慮掃讀效率

## 9. 表單規範
1. 欄位名稱固定，不可同義重複
2. 輸入框、下拉、文字區高度與間距需一致
3. 驗證訊息放欄位附近，不只用 toast
4. 表單頂部需提供 summary，讓使用者知道目前不能儲存的原因
5. 明細表格中的錯誤欄位也要直接標紅，不能只在表格外提示
4. 手機上避免橫向捲動表單

## 10. 後續收斂執行順序
1. badge 系統
2. modal / sticky footer
3. FAB / safe-area
4. Dashboard / Detail / WorkLog 卡片規格
5. 表單欄位與工具列統一


## 11. 本輪已落地規則（V161.291）
1. Quote / Order 狀態 badge 改由核心規則統一取色，避免模組各自漂移
2. 草稿 / 建立 狀態統一回到 primary，不再使用額外自訂紫色語意
3. Sticky footer 補 safe-area 與內容保留空間，避免表單最後欄位被遮住
4. Modal 在手機改為 full-screen sheet 行為，降低窄畫面裁切與誤觸
5. FAB 與底部操作列加入額外避讓，包含 repair mobile actions
6. WorkLog 卡片按鈕在手機改為滿版堆疊，提升現場操作穩定性


## 12. 表單系統收斂規範（V161.292）
1. 表單必須採正式企業級節奏：section → label → control → help / error → action row
2. 不可再使用看起來像 demo、後台測試頁、或只為暫時 patch 拼出的欄位排列
3. Repair / Quote / Order / WorkLog / SOP / Settings 的主要表單需共用同一語言：圓角、層次、欄位高度、提示文字、按鈕節奏
4. 行動裝置優先單欄閱讀；桌機再展開雙欄或三欄
5. 表單說明文字只能做輔助，不能搶過主欄位視覺層級
6. 動作列固定使用 `取消 / 關閉 / 儲存 / 建立 / 更新` 等正式文案，不用實驗性字眼


## 13. 客戶 / 聯絡人表單規範（V161.298）
1. 客戶 / 聯絡人表單需使用真正的 `<form>` submit 結構，不再依賴 `type="button" + data-action` 繞路送出。
2. 公司名稱屬於主欄位，需固定放在第一區塊；聯絡人、電話、Email 按正式聯絡順序排列。
3. 地址與備註必須獨立成補充區，不可與主要聯絡欄位混排。
4. 詳情頁需明確區分：公司 / 聯絡方式 / 維護資訊 / 地址 / 備註。
5. 客戶模組的 footer 需採 sticky footer 規格，手機上按鈕改為滿版堆疊。
6. 客戶表單驗證錯誤需直接落在欄位與表單 summary，不接受只有 toast 的提示方式。

## 客戶 / 聯絡人表單規則（V161.299）
- 主欄位（公司 / 聯絡人 / 電話 / Email）必須優先進入首屏可視區。
- 摘要資訊不可佔用主欄位首屏高度。
- 桌機優先雙欄收斂，手機再回落單欄。
- modal 需採 header / body / sticky footer 分層，不可整體滾動導致表單首屏效率低落。


### 客戶 / 聯絡人表單
- 主欄位（公司、聯絡人、電話、Email）優先保持在主資料段，不可被補充資訊區擠壓。
- 地址與備註屬於次要補充資料，應獨立成段，不做固定窄側欄。
- 摘要列可顯示模式與維修數，但說明文字必須可換行，不得出現截斷。
