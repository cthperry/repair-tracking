🔴 P0 — 高風險 / 立即處理

P0
全域命名空間污染（window.* 蔓延）

所有模組直接掛在 window：window.QuoteModel、window.AuthSystem、window.QuotesUI、window.CustomerUIForms… 估計超過 40 個全域變數。任何第三方腳本、CDN 衝突或打字錯誤都可能造成靜默覆蓋，且難以追蹤來源。

更嚴重的是：UI 模組直接呼叫 QuotesUI.setQuickFilter() 寫在 inline HTML attribute onclick，形成無法被靜態分析工具捕捉的隱性依賴。

建議：逐步引入 ES Module（type="module"），或以 IIFE namespace 包裝（window.App.Quotes = ...）至少建立一層隔離。長遠方向：Vite 打包。
P0
God Files — repairs.ui.js (5,046 行) / quotes.ui.js (2,251 行)

repairs.ui.js 單檔 5,046 行承擔了：列表渲染、詳細頁渲染、表單渲染、事件綁定、狀態管理、PDF 匯出、進度追蹤、多個 Modal 邏輯。違反 Single Responsibility，任何修改都需要在大量無關程式碼中導航。

已知副作用：_domBound flag 跨導覽未重置（本次已修正），說明此類 bug 在大文件中難以察覺。

建議：以功能為單位拆分成 repairs.ui-list.js、repairs.ui-detail.js、repairs.ui-form.js（參考 customers 已有 customers.ui-forms.js 的做法）。
P0
Firebase CDN 單點故障 — SDK 載入不穩

三個 Firebase compat 腳本完全依賴 gstatic.com CDN，無本地備份。本次已重現：CDN 延遲 → typeof firebase === 'undefined' → auth 初始化失敗 → 登入無效。雖然已加入 5 秒 polling，但根本問題未解決。

台灣/亞洲地區 Google CDN 偶發性不穩定是已知現象。

建議：將 Firebase SDK 下載至 /assets/vendor/firebase/ 本地化，消除 CDN 依賴。或使用 npm + Vite 打包時一並解決。

🟡 P1 — 近期規劃

P1
無模組打包工具 — 40+ <script> 串行載入

HTML 中有 40+ 個手動 <script> 標籤，全部串行執行。缺乏 tree-shaking、code splitting、壓縮。BUILD_NUMBER 手動遞增作為 cache-bust，容易遺漏。

建議：引入 Vite（對 vanilla JS 友善，配置簡單）。保留現有架構，改由 Vite 處理 bundle + version hash，移除 BUILD_NUMBER 手動維護。
P1
HTML 字串拼接 — 潛在 XSS 風險

所有 UI 模組使用 Template Literal 直接拼接 HTML：<div>${customer.name}</div>。雖然部分地方有 _escape()、_escapeAttr()，但覆蓋不一致。若 Firebase 資料被污染或有人注入 <script>，前端即遭 XSS。

建議：統一使用 _esc() 套用到所有用戶輸入欄位；或引入輕量模板庫（如 lit-html）自動 escape。
P1
跨模組通信混用兩種機制

模組間通信同時使用：① window.dispatchEvent(CustomEvent)，② linkage.js 的 pub/sub，③ 直接 window.RepairService.xxx() 呼叫。三種方式混用導致依賴關係難以追蹤，特別是在除錯事件傳播問題時。

建議：統一使用 linkage.js 作為唯一跨模組事件通道；禁止模組直接呼叫其他模組的 UI 實例。
P1
測試覆蓋率不明 — 業務核心邏輯風險

框架齊備（Vitest + Playwright），但 repairs.service.js（1,234 行）、quotes.service.js（998 行）等核心業務邏輯的單元測試覆蓋率不明。本次發現的 form.id / HTML named access bug 應可被自動化測試攔截。

建議：執行 npm test -- --coverage 取得覆蓋率報告；優先補齊 repairs.service、quotes.service、auth.js 的測試。目標：業務邏輯 ≥ 70%。

🔵 P2 — 長期改善

P2
無 TypeScript — 運行時錯誤難以預防

所有 JS 皆為純 JavaScript，無型別系統。form.id 回傳 DOM element 而非 string 的 bug 在 TypeScript + strict mode 下可在開發期捕獲。大型 UI 文件中的資料流難以靜態驗證。

建議：可從 JSDoc 型別標注開始（零遷移成本），配合 VS Code 的 TypeScript language server 取得型別提示，無需重寫。
P2
Desktop/Mobile 共用邏輯但 HTML 分離

Desktop 和 Mobile 各有獨立 HTML 入口，部分 CSS（corporate-enhancement.css）只在 Desktop 載入，造成 Mobile 視覺差異。兩份 HTML 的腳本載入清單需手動同步，容易漏載入新模組。

建議：將 CSS 載入移至 module-loader.js 管理，由 JS 根據 viewport 決定載入哪套 CSS；或移除 Mobile 獨立 HTML，統一用 responsive CSS 處理。
P2
Service 層 Firebase 初始化分散

每個 Service 都有自己的 init() 方法獨立連接 Firebase，初始化邏輯重複（snapshot listener 設置、error handling、Taipei timezone counter）。維護時需在 20 個 service 中同步修改。

建議：抽取 BaseFirebaseService 類別，所有 service 繼承它，統一處理 RTDB 連接、snapshot 管理、重試邏輯。

⚠️ UI 問題

P0
響應式斷點不一致 — 各模組標準不統一

僅 quotes.css 有完整的 1200/1024/960/768px 斷點（本次新增）。其他模組：

• repairs.css：只有 768px
• customers.css：只有 768px
• orders.css、parts.css：只有 768px

平板（960-1200px）用戶在維修單、客戶管理模組仍會看到水平捲動。

建議：在 shared.css 或 core/ui.css 建立全域 table 斷點規則（.data-table-wrap），套用至所有模組，避免重複在每個 CSS 設定。
P0
Mobile 版缺少 corporate-enhancement.css

Desktop HTML 載入 corporate-enhancement.css（17KB，包含 sidebar 深色主題、字型精修、元件細節），Mobile HTML 未載入。造成兩版本視覺品質落差：按鈕、卡片、間距等在 Mobile 比 Desktop 明顯粗糙。

建議：將 corporate-enhancement.css 中與 sidebar 無關的通用部分（字型、按鈕、卡片）移至 shared.css；sidebar 專屬樣式保留在 desktop-only 的 CSS 中。
P1
空狀態（Empty State）設計不一致

部分模組的空狀態只顯示「沒有資料」，缺乏說明原因（是過濾條件太嚴，還是資料庫真的空）和引導動作（「立即建立第一張報價單」按鈕）。符合標準的 UX Writing 格式應為：這是什麼 → 為什麼是空的 → 怎麼開始。

建議：區分「篩選無結果」和「真實空狀態」兩種情況，前者顯示「清除篩選條件」按鈕，後者顯示 CTA 進入新增流程。
P1
PDF 匯出無進度回饋

點擊「匯出 PDF」後，前端在字體載入 + pdf-lib 運算期間（可能 2-5 秒）無任何視覺反饋，用戶不知道系統是否在處理中，傾向重複點擊。

建議：在 exportQuotePdf() 開始時將按鈕切換為 loading 狀態（⏳ 產生中... + disabled），完成或失敗後恢復。已有 setLoginButtonState() 可參考。
P1
表單無障礙（Accessibility）缺失

HTML 字串拼接的表單缺少 aria-label、aria-describedby、role 等屬性。圖示按鈕（如刪除、編輯）無 aria-label，僅有 emoji 作為內容，螢幕閱讀器無法識別。錯誤訊息未透過 aria-live 通知輔助技術。

建議：在 core/ui.js 的 createButton()、createInput() 等 factory 函數中加入 aria 參數，確保新元件預設符合 WCAG 2.1 AA。
P2
部分顏色值仍 hardcoded 在非 quotes 模組

本次已完成 quotes.ui.js 的顏色 variable 化。但 repairs.ui.js、maintenance.ui.js、customers.ui.js 等是否也有 hardcoded hex 值（#16a34a 等）尚未全面審查。

建議：執行 grep -r "#[0-9a-f]\{6\}" features/ 找出所有 hardcoded 顏色，逐步替換為 CSS variable 引用。
